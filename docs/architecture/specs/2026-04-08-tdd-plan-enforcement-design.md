# TDD Plan Enforcement Design Spec

## Problem

Implementation plans generated by agents do not follow the TDD cycle defined in AGENTS.md. The existing rule ("ALWAYS write a failing test first, implement until it passes, then commit") is too vague to enforce consistent plan structure. Plans frequently batch implementation steps before tests or omit verify steps entirely.

## Solution

Replace the single TDD rule in AGENTS.md with four specific rules that define both the TDD cycle and the required plan task structure.

## Changes

**File:** `AGENTS.md`

**Remove:**
```
- ALWAYS write a failing test first, implement until it passes, then commit.
```

**Replace with:**
```
- ALWAYS follow the TDD cycle: write a failing test, verify it fails, implement minimal code to pass, verify it passes, then commit.
- ALWAYS structure each plan task as: test -> verify-fail -> implement -> verify-pass -> commit. Each step is one action.
- ALWAYS pair each implementation task with its own test -- write and verify the test before writing any implementation code for that task.
- ALWAYS include exact test commands with expected output (pass or fail) in every verify step.
```

## Rationale

- Four rules instead of one: separates the cycle, plan structure, pairing discipline, and verification concreteness.
- All rules use positive guidance ("ALWAYS do X") per AGENTS.md convention.
- Rules are in AGENTS.md (user-controlled) so they apply regardless of which skill generates plans.
- Directly addresses the gap between "do TDD" and "here's what TDD looks like in a plan step."
